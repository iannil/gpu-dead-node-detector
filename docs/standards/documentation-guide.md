# 文档规范指南

**版本**: 1.0
**创建日期**: 2026-01-18
**维护者**: GDND 开发团队

---

## 1. 文档分类

### 1.1 进展文档 (`/docs/progress/`)

记录进行中的工作，包括：
- 功能开发进度
- 问题排查记录
- 技术调研结果

**状态**: 工作完成后应迁移到 `/docs/reports/completed/`

### 1.2 完成报告 (`/docs/reports/completed/`)

记录已完成的工作，包括：
- 功能实现总结
- 变更详细记录
- 测试结果

### 1.3 验收文档 (`/docs/reports/`)

记录对已完成工作的验收，包括：
- 验收标准
- 验收结果
- 遗留问题

### 1.4 文档规范 (`/docs/standards/`)

记录项目标准和规范

### 1.5 文档模板 (`/docs/templates/`)

提供各类文档的标准模板

---

## 2. 命名规范

### 2.1 进展文档

格式: `YYYY-MM-DD-<简短描述>.md`

示例:
- `2026-01-18-project-init.md`
- `2026-01-20-nvml-integration.md`
- `2026-02-01-helm-chart-design.md`

### 2.2 完成报告

格式: `YYYY-MM-DD-<功能名称>-completed.md`

示例:
- `2026-01-25-l1-detection-completed.md`
- `2026-02-10-kubernetes-integration-completed.md`

### 2.3 验收文档

格式: `YYYY-MM-DD-<功能名称>-review.md`

示例:
- `2026-01-26-l1-detection-review.md`

---

## 3. 文档结构要求

### 3.1 必需元信息

每个文档开头必须包含：

```markdown
# 文档标题

**文档编号**: <类型>-YYYY-MM-DD-<序号>
**创建日期**: YYYY-MM-DD
**最后更新**: YYYY-MM-DD
**状态**: <进行中|已完成|待验收|已验收>
```

文档编号类型：
- `PROG` - 进展文档
- `COMP` - 完成报告
- `REVW` - 验收文档

### 3.2 时间戳记录

对于重要的修改或进展，必须记录具体时间：

```markdown
### 2026-01-18 14:30 - 完成 NVML 绑定

实现了以下功能：
- ...
```

---

## 4. 文档状态流转

```
[新建] → 进行中 → 已完成 → 待验收 → 已验收
                     ↓
              (发现问题)
                     ↓
              返回进行中
```

### 4.1 状态说明

| 状态 | 存放位置 | 说明 |
|------|----------|------|
| 进行中 | `/docs/progress/` | 工作正在进行 |
| 已完成 | `/docs/reports/completed/` | 开发完成，待验收 |
| 待验收 | `/docs/reports/` | 已提交验收文档 |
| 已验收 | `/docs/reports/completed/` | 验收通过 |

---

## 5. 内容规范

### 5.1 语言

- 文档正文使用**中文**
- 代码、命令、配置使用**英文**
- 技术术语可保留英文原文

### 5.2 格式

- 使用 Markdown 格式
- 代码块指定语言类型
- 表格对齐整洁
- 适当使用标题层级（最多4级）

### 5.3 图表

- 架构图使用 Mermaid 语法
- 复杂流程使用流程图说明
- 数据表格使用 Markdown 表格

---

## 6. 维护原则

### 6.1 及时更新

- 进展随时保存，带上实际修改时间
- 完成后立即迁移到对应目录

### 6.2 避免冗余

- 删除过期或无效的内容
- 合并重复的文档
- 保持文档与实际代码同步

### 6.3 可追溯

- 重要变更记录时间戳
- 保留关键决策的原因说明
- 引用相关的 Issue 或 PR 编号
